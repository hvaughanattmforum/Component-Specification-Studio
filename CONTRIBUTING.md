# Contributing to Component Specification Studio

Thanks for your interest in contributing! This project is open to contributions
from anyone. To keep things organized and to make sure changes to `main` are
reviewed consistently, we follow a fork → branch → pull request workflow.

## Before you start: open an issue or discussion

If you're proposing a new feature, a significant change, or aren't sure
whether your idea fits the project, please start a
[Discussion](../../discussions) or open an
[Issue](../../issues) first and describe what you'd like to do.

This lets a maintainer confirm the approach before you invest time in a
branch, and avoids duplicate or conflicting work. Small fixes (typos, obvious
bugs, docs corrections) don't need this step — feel free to jump straight to
a pull request.

## How to contribute

1. **Fork the repository** to your own GitHub account.
2. **Create a branch** in your fork for your change:
   ```bash
   git checkout -b your-feature-name
   ```
3. **Make your changes** and commit them with clear, descriptive messages.
4. **Push your branch** to your fork:
   ```bash
   git push origin your-feature-name
   ```
5. **Open a pull request** from your branch into this repository's `main`
   branch. Reference the related issue or discussion in the PR description.

You don't need write access to this repository to do any of the above —
forking and opening a pull request is open to everyone.

## Review and merging

- Every pull request requires review and approval from a code owner
  (see [CODEOWNERS](CODEOWNERS)) before it can be merged.
- Only `@hvaughanattmforum` can approve and merge changes into `main`.
- A maintainer may ask for changes, request tests, or discuss the approach
  in the PR thread — please be responsive to review feedback.
- Once approved, the maintainer will merge the pull request.

## Guidelines for a good pull request

- Keep changes focused — one logical change per PR is easier to review.
- Describe **what** the change does and **why** it's needed.
- Update or add documentation if your change affects usage or behavior.
- Make sure the project builds and any existing checks pass before
  requesting review.

## Questions?

If anything here is unclear, open a
[Discussion](../../discussions) and a maintainer will help.
